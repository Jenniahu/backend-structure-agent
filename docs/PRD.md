# 后端架构学习 Agent —— 产品设计文档 (PRD)

> 版本: v0.2 (MVP)
> 文档状态: 待评审
> 一句话定位: 一个用「讲解 + 问答 + 作业 + 结构化反馈」闭环,帮有技术基础但缺工程经验的计算机学生建立后端架构思维、并能应对架构面试的个性化 AI 学习平台。
> 技术栈: Hono + Node.js + PostgreSQL,部署目标 Render,含简单邮箱密码登录。
>
> 变更记录:
> - v0.2: 后端改为 Hono + Node.js + PostgreSQL(原 Cloudflare + D1),部署目标改为 Render;新增简单登录(邮箱+密码)及对应数据模型与 API;新增 Render 部署方案。

## 1. 背景与问题

### 1.1 目标人群
在校 / 应届计算机专业学生,特征:
- 懂数据结构、算法、基础数据库与网络;
- 知道「缓存 / 消息队列 / 分库分表」这些词,但**不知道什么时候用、为什么用、不用会怎样**;
- 缺乏真实高并发系统经验,说不出技术决策背后的「取舍」;
- 面试常被架构题、系统设计题考倒,尤其是被「追问」时。

### 1.2 真实痛点(表层 vs 本质)
| 表层需求 | 真实需求 |
|---------|---------|
| 帮我设计后端架构 | 我能在面试中**讲清楚**一套架构的取舍逻辑 |
| 我缺工程经验 | 我没踩过坑,所以**说不出「为什么这么选」** |

### 1.3 核心洞察
架构面试考的不是「图画得对不对」,而是:
1. 需求澄清能力;2. trade-off(权衡)意识;3. 容量/QPS 估算能力;4. 应对追问的深度与应变。
因此本产品的核心价值 **不是给标准答案,而是训练架构思维 + 提供个性化陪练与反馈**。

> ⚠️ 反目标:绝不做成「输入需求 → 输出架构图」的填鸭工具,避免用户变成背答案、一追问就崩。

## 2. 产品理念:学练用闭环

```
        ┌──────────────────────────────────────────┐
        │                                          │
        ▼                                          │
   ① 讲解(Teach) → ② 问答(Check) → ③ 作业(Apply) → ④ 反馈(Feedback)
   引入概念+类比     苏格拉底追问      动手设计任务       结构化打分+诊断
        │                                          ▲
        └──────────────────────────────────────────┘
              做不好 → 归因薄弱点 → 回到讲解重讲 → 再练
```

四个环节是**循环**而非线性,作业暴露的薄弱点会驱动回炉重学。这是区别于「填鸭」的本质设计。

### 2.1 四环节设计要点
- **① 讲解 Teach**: 分层、可中断、场景钩子引入;每讲 3-5 句必抛一个问题;善用类比降门槛(缓存=书桌常用书,DB=图书馆)。**铁律:绝不连续输出超过一屏。**
- **② 问答 Check**: 苏格拉底式。答对→追问更深一层;答错→反问引导而非直接纠正;答不上→回退换角度重讲。目的是消灭「假懂」。
- **③ 作业 Apply**: 梯度任务(见 5.4),从判断题到完整系统设计。
- **④ 反馈 Feedback**: 5 维度结构化打分(见 5.5),指出做得好/问题/缺失,并给追问挑战与下一步建议。

### 2.2 贯穿始终的方法论脚手架
所有讲解、问答、作业都围绕这套通用系统设计框架展开,让用户逐步内化为「肌肉记忆」:
```
需求澄清 → 容量估算 → 接口设计 → 数据模型 → 高层架构 → 深挖瓶颈 → 权衡讨论
```

### 2.3 「不填鸭」的五条原则
1. 能问就不讲(优先提问引导);2. 小步快跑(讲一点问一点练一点);3. 进度可视化;4. 错题本机制;5. 一切从场景出发。

## 3. Agent 在平台中的角色

**平台是壳,Agent(LLM)是大脑。** 前端只负责展示与交互,所有「判断、讲解、批改、评估、追问」都由后端调用 LLM 完成。

| # | 场景 | Agent 职责 | 是否硬依赖 LLM |
|---|------|-----------|--------------|
| 1 | 讲解 | 按用户水平/偏好动态生成讲解 | 可静态可动态 |
| 2 | 互动问答 | 判断回答对错,决定追问或回退 | ✅ 必须 |
| 3 | 作业批改 | 接收设计文字,输出结构化打分+点评 | ✅ 核心 |
| 4 | 掌握度评估 | 根据表现评估知识点掌握程度 | ✅ 必须 |
| 5 | 错题归因 | 分析错误类别,生成复习建议 | ✅ 必须 |

### 3.1 典型数据流
```
用户在网页输入作业答案
   → 前端 POST /api/grade
   → 后端召回用户长期记忆 + 拼装「批改 Prompt」+ 注入作业 rubric
   → 调用 LLM(OpenAI 兼容接口)
   → LLM 返回结构化打分 JSON
   → 后端存入 D1(更新进度/错题/掌握度) + 返回前端展示
```

## 4. 记忆系统设计(短期 + 长期)

参考 Agent harness 的短/长期记忆架构。这是平台「个性化」的核心。

### 4.1 短期记忆 (Session)
- 范围: 当前一次学习/作业会话;
- 内容: 本次对话历史、当前作业上下文、刚答错的题;
- 存储: 会话表 / 内存,会话结束可清理。

### 4.2 长期记忆 (Profile,跨会话持久化)
三类画像,存于 D1:

| 类别 | 内容 | 用途 |
|------|------|------|
| ① 学习偏好画像 Preference | 讲解风格(类比/直接/图示)、节奏(快讲/细嚼)、偏好的学习顺序 | 个性化讲解风格与节奏 |
| ② 能力画像 Mastery | 每个知识点掌握度评分、状态(未掌握/学习中/已掌握) | 课程地图状态、解锁、推荐复习 |
| ③ 错误档案 Error | 高频错误点、错误归类(需求澄清弱/取舍弱/估算弱…) | 错题本、批改时一针见血、针对性复练 |

### 4.3 记忆飞轮(协同机制)
```
召回 (新会话开始,从 D1 取出用户画像注入 Prompt)
  → 个性化 (按偏好讲解,按薄弱点重点提醒)
  → 沉淀 (会话/批改后,Agent 产出「本次观察」更新画像)
  → 再召回 (下次会话用更新后的画像)
```
- 召回示例注入: 「该用户偏好类比讲解;容量估算为弱项,讲解与批改时请重点关注。」
- 沉淀示例: 批改后 Agent 输出 `{ "weakness_observed": "容量估算缺失", "mastery_delta": {"cache": +5} }`,后端据此更新长期记忆。

## 5. 功能规格

### 5.1 课程地图(侧边栏)
- 展示主题列表与状态(✅已掌握 / 🔄学习中 / 🔒未解锁);
- 展示掌握度、错题本入口;
- 状态由长期记忆(Mastery)驱动。

### 5.2 学习主区(三种交互形态)
- 讲解卡片(分层、可点「继续/换种讲法/我没懂」);
- 互动问答(输入回答 → Agent 判定 → 追问/回退);
- 作业区(展示题目 + 输入框 → 提交 → 结构化反馈报告)。

### 5.3 首批内容范围(MVP)
- 主题: `缓存` `消息队列` `数据库选型` `负载均衡` `分库分表`
- 案例: `短链系统` `Feed 流` `秒杀`
- **MVP 先跑通「缓存」一个主题的完整闭环**(讲解→问答→作业→打分),验证体验后再批量补内容。

### 5.4 作业难度梯度
| 难度 | 形式 | 例子 |
|------|------|------|
| 🟢 入门 | 判断/选择+说理由 | 三个场景哪个该用缓存?为什么? |
| 🟡 进阶 | 局部设计 | 给短链系统设计存储表结构 |
| 🔴 综合 | 完整系统设计 | 设计支持百万 DAU 的点赞系统 |
| 🔥 挑战 | 带约束/反常识 | 不加机器把接口 QPS 提升 10 倍 |

### 5.5 结构化反馈(5 维度 rubric)
```
📋 作业反馈报告
─────────────────────────────────────
总分: 62 / 100   等级: B-(有思路,深度不足)

① 需求澄清    ███████░░░  7/10
   ✅ 你识别了读多写少
   ⚠️ 没问清一致性要求(强一致 or 最终一致?)
② 容量估算    ████░░░░░░  4/10
   ❌ 完全没估算 QPS 和存储量,无法判断选型
③ 架构合理性  ██████░░░░  6/10
   ✅ 缓存+DB 分层正确    ⚠️ 缓存击穿没处理
④ 取舍意识    █████░░░░░  5/10
   ⚠️ 只说了用什么,没说为什么不用别的
⑤ 表达清晰度  ████████░░  8/10
─────────────────────────────────────
💡 下一步建议:重点补「容量估算」,去学习模块复习
🔥 追问挑战:如果热点 key 同时过期,你怎么防雪崩?
```
- 5 维度: 需求澄清 / 容量估算 / 架构合理性 / 取舍意识 / 表达清晰度;
- 每个主题的作业配独立 rubric 文件,保证打分标准一致(而非 LLM 随机发挥)。

## 6. 知识库设计

### 6.1 原则:不爬技术文档,沉淀教学内容
- ❌ 不抓 MySQL/Redis/Kafka 官方大部头文档(LLM 已掌握、易稀释重点、易过时);
- ✅ 沉淀「教学结构化内容」,这才是产品壁垒。

### 6.2 知识库四类内容
| 类型 | 内容 | 为何要自建 |
|------|------|-----------|
| 学习路径/大纲 | 主题顺序、知识点拆解、解锁条件 | LLM 不知道你想怎么教 |
| 经典案例库 | 短链/Feed流/秒杀 标准拆解 | 保证讲解质量稳定 |
| 作业题库+评分标准 | 每主题 3-4 道分难度作业 + rubric | 打分一致性关键 |
| 反模式/常见错误清单 | 过度设计、忽略一致性等 | 批改一针见血 |

### 6.3 MVP 存储形态(不上 RAG/向量库)
```
knowledge/
├── topics.json          # 学习路径:主题、顺序、解锁条件
├── topics/
│   ├── cache.md         # 缓存:讲解要点+类比+关键问答
│   ├── mq.md
│   └── sharding.md
├── cases/
│   ├── short-url.md
│   └── feed.md
└── assignments/
    └── cache.json       # 缓存作业题 + 评分 rubric
```
后端调 LLM 时把对应 md/json 作为上下文塞进 Prompt。内容量增长到几十上百主题后再考虑 RAG。

## 7. 技术架构(MVP)

> 部署目标平台: **Render**(传统长驻服务器架构,而非边缘 Serverless)。选择理由: 本产品需频繁写入(每次问答/批改都更新记忆)、需要简单登录与会话,传统服务器 + 关系数据库更顺;且教学产品本身讲后端架构,用 PostgreSQL 更名正言顺。

- 前端: 静态页面 + Tailwind(CDN)+ 轻量 JS,无重型框架;
- 后端: **Hono + Node.js**(`@hono/node-server`),作为长驻服务运行;
- LLM: OpenAI 兼容接口(用户已有 API Key,存于**后端环境变量 / Render 环境变量**,**绝不暴露前端**);
- 存储: **PostgreSQL**(关系型数据库)存用户账号/画像/进度/错题/会话;
  - 开发期: sandbox 本地 Postgres(连接串示例 `postgresql://archlearn:****@127.0.0.1:5432/archlearn`);
  - 生产: Render 自带 PostgreSQL(部署时换连接串即可,代码不变);
- 数据库访问: 使用 `pg`(node-postgres)+ 迁移脚本(`migrations/*.sql`);
- 知识库: 仓库内 md/json 文件作为 Prompt 上下文。

### 7.0 为什么是 Render + Postgres(而非 Cloudflare + D1)
| 维度 | Render + Postgres(已选) | Cloudflare + D1 |
|------|--------------------------|-----------------|
| 运行模式 | 传统长驻服务器,能力完整 | 边缘 Serverless,有 CPU/能力限制 |
| 写入体验 | Postgres 工业级,频繁写无压力 | SQLite,并发写弱 |
| 登录/会话 | 传统服务器模型天然支持 | 需额外设计 |
| 数据库成熟度 | PostgreSQL,功能完整稳定 | D1(SQLite),功能较少 |
| 与产品契合 | 讲后端架构用 Postgres 最对味 | — |

### 7.1 核心 API(草案)
| 方法 | 路径 | 作用 | 需登录 |
|------|------|------|------|
| POST | /api/auth/register | 注册(邮箱+密码) | 否 |
| POST | /api/auth/login | 登录,返回会话凭证 | 否 |
| POST | /api/auth/logout | 退出登录 | 是 |
| GET  | /api/auth/me | 获取当前登录用户 | 是 |
| POST | /api/chat | 讲解 + 互动问答(注入记忆与知识) | 是 |
| POST | /api/grade | 作业批改,返回结构化打分 JSON | 是 |
| GET  | /api/profile | 拉取用户长期记忆/进度 | 是 |
| GET  | /api/topics | 课程地图与状态 | 是 |
| POST | /api/memory/update | 沉淀本次观察(更新画像) | 是 |

### 7.2 PostgreSQL 数据模型(草案)
- `users(id, email UNIQUE, password_hash, created_at)` —— 账号(简单登录)
- `auth_sessions(id, user_id, token, expires_at, created_at)` —— 登录会话/凭证
- `preferences(user_id, teach_style, pace, ...)` —— 偏好画像(长期记忆①)
- `mastery(user_id, topic, score, status, updated_at)` —— 能力画像(长期记忆②)
- `errors(user_id, category, topic, count, last_seen)` —— 错误档案(长期记忆③)
- `assignments_history(id, user_id, topic, answer, score_json JSONB, created_at)` —— 作业历史
- `learn_sessions(id, user_id, topic, messages_json JSONB, created_at)` —— 短期会话

> 说明: Postgres 原生支持 `JSONB`,打分结果、对话历史直接以 JSONB 存储更灵活。

### 7.3 简单登录设计(MVP)
- 方式: 邮箱 + 密码;密码用 `bcrypt`(或 Web Crypto)哈希存储,**绝不明文**;
- 会话: 登录后下发随机 token,存 `auth_sessions` 表,前端以 HttpOnly Cookie 持有;
- 鉴权: 受保护接口校验 Cookie 中的 token 是否有效且未过期;
- 范围: MVP 只做注册/登录/登出/获取当前用户,不做第三方 OAuth、找回密码等。

### 7.4 安全约束
- LLM API Key 仅存于后端环境变量(本地 `.env` / 生产 Render 环境变量),所有 LLM 调用走后端,不在前端暴露任何密钥;
- `.env` 必须加入 `.gitignore`,绝不提交;
- 不在前端硬编码任何 token;密码必须哈希存储。

## 8. MVP 范围与验收

### 8.1 MVP 必做
1. **简单登录**:邮箱+密码注册/登录/登出,基于 Postgres + 会话 token;
2. 平台框架(课程地图 + 学习主区);
3. 「缓存」主题完整闭环:讲解 → 互动问答 → 作业 → 5 维度结构化打分;
4. 长期记忆最小版(存于 Postgres):记录缓存主题掌握度 + 高频错误点,登录后新会话能召回;
5. 知识库:`topics.json` + `cache.md` + `cache.json`(作业+rubric);
6. 部署就绪:提供 `render.yaml` + 部署教程,本地用 PM2 跑通验证。

### 8.2 验收标准
- 用户能注册/登录,登录态在刷新后保持;
- 用户能完整走通缓存主题的学练用闭环;
- 作业提交后返回稳定、结构化的 5 维度打分;
- 退出后重新登录,平台能根据上次薄弱点给出针对性提醒(长期记忆生效);
- 全程 LLM Key 不暴露前端,密码不明文存储;
- 代码可在本地用 PM2 跑通,且具备一键部署到 Render 的配置。

### 8.3 后续迭代(非 MVP)
- 补齐其余主题与案例;模拟面试模式;进度可视化报表;错题本针对性复练流;内容多到一定量级后引入 RAG。

## 9. 部署方案(Render)

> 代码由我负责编写并在 sandbox 本地用 PM2 跑通验证;**最后一步部署由你在 Render 控制台点击完成**(本环境无法直接部署到 Render)。

1. 代码 push 到 GitHub;
2. Render 控制台 New → Web Service,连接该仓库;
3. Render 控制台 New → PostgreSQL,创建托管数据库,复制 Internal Database URL;
4. 在 Web Service 环境变量中配置:`DATABASE_URL`(填上一步的连接串)、`OPENAI_API_KEY`、`SESSION_SECRET` 等;
5. 构建命令 / 启动命令由 `render.yaml` 指定(或控制台手填),首次启动自动跑数据库迁移;
6. 我会在 README 写出傻瓜式分步教程。

本地开发连接串(已跑通): `postgresql://archlearn:archlearn_dev@127.0.0.1:5432/archlearn`

## 10. 待确认 / 风险
- LLM 调用成本与速率(批改 Prompt 较长,需控制 token);
- 打分稳定性(靠 rubric + 低 temperature + 结构化输出约束缓解);
- 简单登录的安全性(MVP 不做找回密码/2FA,生产可后续增强);
- sandbox 重启后本地 Postgres 需重新启动(开发期注意,不影响生产)。
