教学引擎改造设计

Phase-based Teaching Engine + 作业前置引导

1背景与问题

ArchLearn 是一个后端架构学习平台，核心学习闭环为：

学习问答
→
作业练习
→
AI 批改打分
→
长期记忆沉淀

当前存在以下问题：

⚠️
教学内容与评分标准严重脱节 学习问答阶段只做概念讲解，从不教「需求澄清 → 容量估算 → 架构设计 → 取舍分析」的系统设计框架； 但作业评分标准对所有难度都按此框架严格评判，导致新人按直觉作答时得分极低（典型案例：入门题 36/100）。
问题具体表现根因
框架未传授学生不知道需要「需求澄清→容量估算」Prompt 只要求讲概念，无教学剧本
AI 行为漂移多轮对话后 AI 忘记教学目标，随机发散纯 Prompt 无结构约束
无进度感学生不知道学到哪了，何时「学完」无状态追踪机制
学习作业割裂作业题和学过的内容没有呼应两个模块完全独立

2设计目标

  1. 让学习问答模块主动教授系统设计思维框架，而不只是概念讲解
  2. 作业答题前给出与学习内容呼应的结构化引导（方案 A）
  3. 用 Spec 文档 + Phase 状态追踪约束 AI 教学行为，避免纯 Prompt 的漂移问题
  4. Phase 推进采用 AI 建议 + 学生确认，保留学习节奏自主权

3.1 Phase 类型词汇表（全局）

定义教学动作的「词汇」，所有主题共用。每个 Phase 类型规定了 AI 的目标和行为模式：

ANCHOR
锚定感知

建立真实感知，激活已有经验。问学生生活中见过什么，不急于灌输概念。

CONFLICT
暴露矛盾

用反例或极端场景让学生感受到「不用这个技术会怎样」，制造认知冲突。

FRAMEWORK
框架植入

主动讲授系统设计框架，但用苏格拉底式问答引导，不直接灌输。

DRILL
框架演练

出迷你场景让学生现场套用框架，AI 给即时反馈，建立肌肉记忆。

CONNECT
作业连接

预告作业考什么，建立学习与作业的心理连接，降低作答时的焦虑感。

3.3 Phase 状态追踪（后端数据库）

新增 learning_phase_state 表，持久化记录用户在每个主题下的教学进度，支持跨会话继续学习。

SQLCREATE TABLE IF NOT EXISTS learning_phase_state (
  user_id           BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  topic_id          TEXT NOT NULL,
  current_phase     TEXT NOT NULL DEFAULT 'ANCHOR',
  completed_phases  TEXT[] NOT NULL DEFAULT '{}',
  turns_in_phase    INTEGER NOT NULL DEFAULT 0,
  phase_started_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, topic_id)
);

4Phase 推进机制

4.1 推进方式：AI 建议 + 学生确认

不自动推进，也不强制推进。由 AI 判断当前 Phase 目标已达成后，在回复末尾附上隐藏标记 [PHASE_READY_TO_ADVANCE]，后端检测到后，前端展示确认按钮：

💡
AI 建议话术示例 「我觉得你已经理解了缓存的核心矛盾，我们进入下一步——框架植入？还是你想再聊聊别的？」
AI 回复含 [PHASE_READY_TO_ADVANCE]
→
后端过滤标记，设 canAdvance: true
→
前端显示「进入下一阶段？」确认按钮
用户点击确认
→
POST /api/phase/:topicId/advance
→
DB 更新 completed_phases + current_phase

4.2 跳题处理：吸收推进原则

学生在早期 Phase 提前问了后期才讲的内容时，AI 不强行拦截，而是：

情况AI 行为
问题与当前 Phase 相关正常回答，继续当前 Phase 目标
问题属于下一个 Phase 的内容简短回答 → 顺势说「你问到这个很好，这正是我们下一步要深入的，稍微提前一点说...」→ 视情况推进 Phase
问题完全偏离主题温和拉回：「这个问题很有意思，我们当前专注在缓存主题，等这个学完我们可以聊那个...」

4.3 长期不推进处理

当 turns_in_phase > maxTurns 时，AI 在回复末尾主动建议推进，并附上 [PHASE_READY_TO_ADVANCE] 标记，避免学生卡死在一个阶段。此规则写入 Prompt 规范，由 AI 自行执行。

5Prompt 设计规范

每次调用 AI 时，在现有 system prompt 中追加以下结构化 context 块，注入在主题素材之后、用户记忆画像之前：

System Prompt 注入模板【当前教学阶段】
- 已完成阶段：{completedPhases}（如 ["ANCHOR", "CONFLICT"]）
- 当前阶段：{currentPhase}（{phaseTypeDescription}）
- 当前阶段目标：{phaseGoal}
- 成功信号：{successSignal}
- 已在本阶段对话 {turnsInPhase} 轮（上限 {maxTurns} 轮）

【本阶段行为规则——非常重要】
1. 你的首要任务是达成当前阶段目标，不要偏离
2. 成功信号达成后，在回复末尾加上标记 [PHASE_READY_TO_ADVANCE]
3. 如果 turnsInPhase >= maxTurns，无论是否达成，也主动建议推进并附上标记
4. 如果学生提前问了后续阶段的内容，简短回答后顺势引导，可以适当提前推进
5. 当前处于 {currentPhase} 阶段，不要完整展开 {futurePhases} 阶段的内容
⚠️
标记处理规则 后端检测到 [PHASE_READY_TO_ADVANCE] 后，必须从展示给用户的文本中清除该标记，仅通过响应字段 canAdvance: true 传递给前端。

6前端：作业前置引导（方案 A）

当学生点击某道作业题时，在答题框上方展示动态引导提示，内容来自 Spec 文档的 assignmentFrameworkHints 字段。

UI 规格

元素说明
展示方式折叠卡片，默认展开；标题「📋 答题框架提示」
内容来源从 GET /api/topics/:id/assignments 的 frameworkHints 字段读取
状态联动已完成 FRAMEWORK Phase → 提示前加「✅ 你在学习中已练习过这些框架」
未完成状态提示前加「💡 建议先完成学习问答中的框架讲解再作答」

连接逻辑

提示内容从 Spec 文档派生（单一事实来源），与学习阶段状态联动，保证学习和作业的一致性。前端通过比对 GET /api/phase/:topicId 返回的 completedPhases 判断是否已完成 FRAMEWORK 阶段。

10API 设计

10.1 认证方式

所有业务接口（/api/* 除健康检查外）均需登录鉴权，鉴权方式为 Bearer Token：

Authorization: Bearer <token>

Token 由 POST /api/auth/login 颁发，存储在 auth_sessions 表中，服务端每次请求校验 token 有效性及过期时间（expires_at）。前端将 token 存入 localStorage，通过 requireAuth 中间件统一拦截。

10.2 API 总览

// routes/api.ts
apiRoutes.post('/chat', async (c) => {
  const user = currentUser(c)
  const { topicId, history } = await c.req.json()

  const mem = await recallMemory(user.id, topicId)
  const msgs = Array.isArray(history) ? history : []

  // 新增：读取 Phase 状态 + 主题 Spec
  const phaseState = await getOrInitPhaseState(
    user.id, topicId
  )
  const spec = getTopicSpec(topicId)

  // 注入 Phase context
  const rawReply = await teachChat(
    topicId, msgs, mem, phaseState, spec
  )

  // 检测推进标记
  const canAdvance = rawReply.includes(
    '[PHASE_READY_TO_ADVANCE]'
  )
  const reply = rawReply
    .replace('[PHASE_READY_TO_ADVANCE]', '')
    .trim()

  // 更新轮数
  await incrementTurns(user.id, topicId)

  return c.json({ reply, canAdvance })
})

  "canAdvance": true   // ← 新增字段
