# 缓存(Cache)教学知识卡

> 本文件是喂给讲解/问答 Agent 的「教学素材与边界」。Agent 必须基于此内容讲解,保证质量稳定,而非自由发挥。

## 一、本主题的教学目标
学生学完后应能回答:
1. 缓存解决什么问题?(读多写少 → 降压、降延迟)
2. 缓存的代价是什么?(一致性问题)
3. 三大经典问题(穿透/击穿/雪崩)分别是什么、怎么防?
4. 什么时候【不】该用缓存?

## 二、场景钩子(用于引入,不要一上来讲定义)
> "假设你做了个论坛,平时没人,突然一篇帖子上了热搜,10 万人同时刷新首页。你的数据库 QPS 瞬间飙到几万,直接被打挂了。在不无限加数据库的前提下,你会怎么办?"

引导学生想到:大量请求其实在读【同样的数据】→ 没必要每次都查数据库 → 把热点数据放到更快的地方 → 这就是缓存。

## 三、核心类比(降低门槛)
- **缓存 = 你书桌上的常用书**;**数据库 = 图书馆**。
- 每次查资料都跑图书馆(数据库)太慢;把常用的几本放书桌上(缓存),拿起来就用。
- 书桌空间有限 → 缓存容量有限 → 需要淘汰策略(LRU)。
- 图书馆的书更新了,但你书桌上的是旧版 → **缓存一致性问题**。

## 四、关键知识点(讲解时分小步,每步后提问)

### 1. 缓存为什么快
内存读取 vs 磁盘/网络查询,差几个数量级。缓存(如 Redis)在内存,数据库要读磁盘+复杂查询。

### 2. 适用场景:读多写少
- ✅ 适合:热点商品详情、用户资料、配置、首页内容
- ❌ 不适合:写极频繁的数据(缓存频繁失效,得不偿失)、强一致要求的数据(如账户余额扣减)

### 3. 缓存更新策略(取舍)
- **Cache-Aside(旁路,最常用)**:读时先查缓存,没有再查 DB 并回填;写时更新 DB 并删除缓存。简单但有短暂不一致窗口。
- **Write-Through**:写时同时写缓存和 DB。一致性好,写入慢。
- **Write-Back**:先写缓存,异步刷 DB。最快,但宕机可能丢数据。
> 教学重点:让学生说出每种策略的【代价】,而不是背名字。

### 4. 三大经典问题(面试高频)
- **缓存穿透**:查一个【根本不存在】的数据,缓存和 DB 都没有,每次都打到 DB。
  - 防:布隆过滤器 / 缓存空值。
- **缓存击穿**:某个【热点 key】突然过期的瞬间,大量请求同时打到 DB。
  - 防:互斥锁(只让一个请求去回源)/ 热点 key 不过期。
- **缓存雪崩**:大量 key【同时过期】或缓存整体宕机,流量瞬间全压到 DB。
  - 防:过期时间加随机抖动 / 缓存高可用 / 限流降级。
> 记忆口诀:穿透=查不存在的;击穿=单个热点过期;雪崩=大面积同时失效。

### 5. 容量估算的引导(本主题要刻意训练)
引导学生估算:比如缓存 100 万个用户资料,每个 1KB → 约 1GB 内存。让他建立"这个缓存要多大、一台 Redis 够不够"的量级直觉。

## 五、常见误区(讲解/批改时要警惕学生犯)
- ❌ 一上来就"加缓存",不分析是不是读多写少
- ❌ 只说"用 Redis",说不出更新策略和一致性代价
- ❌ 忽略缓存穿透/击穿/雪崩
- ❌ 强一致场景(如扣款)也想用缓存
- ❌ 不做容量估算,不知道缓存要多大

## 六、苏格拉底式追问清单(供问答模式调用)
- "你说加缓存,那缓存里的数据和数据库不一致了怎么办?"
- "如果有人专门查不存在的 id,缓存能挡住吗?"
- "热点 key 过期那一瞬间会发生什么?"
- "如果所有 key 都设了 1 小时过期,1 小时后会怎样?"
- "这个缓存大概要多大内存?你算过吗?"
